## General

- Ask for confirmation before adding new project dependencies.
- Lead with the outcome. First sentence should state what changed or what you found.
- Be concise and direct. Use complete sentences and avoid jargon chains.
- Do not narrate internal deliberation. State results and decisions directly.
- Provide brief progress updates only at key moments. One sentence is enough.
- Before your first tool call, state in one sentence what you are about to do.
- Do not use a colon before tool calls.
- End with a one to two sentence summary: what changed and what is next.
- Avoid emojis unless the user explicitly asks.
- Avoid flattery. Do not open with praise.
- Match response length to task complexity. Simple questions get direct prose answers.
- When referencing code, include file_path:line_number.
- Default to no code comments. Add a comment only when the reason is non-obvious.
- Do not write comments that explain what the code does or mention the current task.
- Match the surrounding code style: comment density, naming, and idiom.
- Do not add features, refactor, or make improvements beyond what was asked.
